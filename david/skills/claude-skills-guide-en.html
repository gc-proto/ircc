Writing Claude Code skills for GC web content

Claude Code skills are reusable sets of instructions stored as Markdown files that extend what Claude Code can do. They allow you to teach Claude your team's standards, workflows, and patterns for creating Government of Canada (GC) web content that follows Canada.ca guidelines, the GCWeb theme, and WET-BOEW conventions.

On this page

  • File structure

What are Claude Code skills

A Claude Code skill is a directory inside .claude/skills/ that contains a SKILL.md file with YAML frontmatter and Markdown instructions. Skills let you:

  • define reusable workflows Claude follows when invoked
  • provide reference knowledge Claude loads automatically when relevant
  • enforce coding standards, accessibility requirements, and content guidelines
  • include supporting files such as templates, examples, and scripts

Skills are invoked as slash commands (for example, /gc-validate) or loaded automatically by Claude when your prompt matches the skill's description.

File structure

Skills live inside a .claude/skills/ directory in your project or your home directory. Each skill is a folder containing at minimum a SKILL.md file.

Basic layout

.claude/
  skills/
    gc-page-builder/
      SKILL.md              # Required: instructions and configuration
      reference/
        wet-boew-components.md  # Optional: supporting docs
      templates/
        basic-page-en.html      # Optional: template files
      examples/
        valid-page.html         # Optional: example files
    gc-content-validator/
      SKILL.md
    a11y-audit/
      SKILL.md

Where skills can live

Skill scope and precedence
Location Path Applies to
Personal ~/.claude/skills/<name>/SKILL.md All your projects
Project .claude/skills/<name>/SKILL.md Current project only

When skills share the same name, personal skills take precedence over project skills.

How to create a skill

Follow these steps to create a new skill.

Step 1: Create the skill directory

In your terminal, create the folder structure:

mkdir -p .claude/skills/my-skill-name

Step 2: Create SKILL.md

Create a file called SKILL.md inside the skill directory. The file has two parts: YAML frontmatter (configuration) at the top, and Markdown instructions (what Claude should do) below.

---
name: my-skill-name
description: Brief description of what this skill does and when to use it.
---

## Instructions

Write your instructions for Claude here.
Claude will follow these instructions when the skill is invoked.

Step 3: Add supporting files (optional)

For complex skills, add reference documents, templates, or scripts in sub-folders. Reference them from SKILL.md so Claude loads them only when needed:

For component details, see [wet-boew-components.md](reference/wet-boew-components.md).

Step 4: Test the skill

In Claude Code, type /my-skill-name to invoke it directly and verify it works as expected.

Skill configuration (frontmatter)

The YAML frontmatter at the top of SKILL.md controls how the skill behaves.

SKILL.md frontmatter fields
Field Required Description
name No Lowercase name with hyphens (max 64 characters). Defaults to directory name.
description Recommended Claude uses this to decide when to load the skill automatically. Be specific.
disable-model-invocation No Set to true for skills with side effects (deploy, commit). Prevents Claude from invoking the skill on its own.
user-invocable No Set to false for background knowledge skills that only Claude can use.
argument-hint No Hint text shown during autocomplete, for example [filename].
allowed-tools No Tools available without per-use permission, for example Read, Grep, Glob.
context No Set to fork to run in an isolated subagent.

How skills are invoked

There are two ways skills get loaded.

Direct invocation

Type a slash command in Claude Code:

/gc-validate index-en.html
/gc-page-builder
/a11y-audit

Arguments are available inside the skill as $ARGUMENTS (full string) or $0, $1, $2 (individual arguments).

Automatic invocation

Claude reads the description field in your skill's frontmatter. If your prompt matches, Claude loads the skill automatically. For example, if your skill description mentions "Canada.ca accessibility" and you ask Claude to "check this page for accessibility issues," Claude loads the skill without you typing a slash command.

Tip

Write specific, keyword-rich descriptions so Claude matches the right skill. A description like "Validate HTML against Canada.ca WET-BOEW standards and WCAG 2.0 Level AA accessibility requirements" works better than "check web pages."

Example skills for GC web content

The following examples show skills tailored for creating and maintaining Government of Canada web content using the GCWeb theme and WET-BOEW framework.

  • GC page builder: create new Canada.ca pages from a template

    This skill generates new GC web pages using your project's HTML template, ensuring proper structure, bilingual metadata, and WET-BOEW components.

    Create the file .claude/skills/gc-page-builder/SKILL.md:

    ---
name: gc-page-builder
description: Create new Canada.ca web pages using the GCWeb theme and WET-BOEW framework. Use when building new HTML pages for Government of Canada content.
argument-hint: "[page-title]"
disable-model-invocation: true
---

## Create a new GC web page

Build a new Canada.ca page titled "$ARGUMENTS" using the basic template.

### Requirements

1. Use the template at templates/template-basic-en.html as the base
2. Set the page title in the title tag and h1
3. Include proper breadcrumb navigation
4. Add an "On this page" section linking to all h2 headings
5. Use only WET-BOEW and GCWeb approved components:
   - Bootstrap 3 grid (container, row, col-*)
   - WET-BOEW expand/collapse (details/summary)
   - GC responsive tables (class="provisional gc-table table")
   - Alert boxes (alert-info, alert-warning, alert-danger, alert-success)
6. Set the date modified to today's date
7. Ensure the page validates against WCAG 2.0 Level AA:
   - Proper heading hierarchy (h1, h2, h3 in order)
   - Descriptive link text (not "click here")
   - Alt text on all images
   - Proper lang attributes
  • GC content validator: check pages against Canada.ca standards

    This skill audits an existing HTML page against Canada.ca content and accessibility standards.

    Create the file .claude/skills/gc-content-validator/SKILL.md:

    ---
name: gc-content-validator
description: Validate HTML content against Canada.ca standards, WET-BOEW components, and WCAG 2.0 Level AA accessibility. Use when reviewing or auditing GC web pages.
argument-hint: "[filename]"
allowed-tools: Read, Grep, Glob
---

## Validate GC web content

Read and validate the file $ARGUMENTS against these standards:

### Structure checks
- Page uses the GCWeb theme CSS and JS from wet-boew.github.io
- Has skip navigation links (Skip to main content)
- Contains proper breadcrumb navigation
- Has a single h1 inside the main element
- Heading hierarchy has no gaps (no h1 followed by h3)
- Date modified is present and in YYYY-MM-DD format
- Footer contains Government of Canada branding

### Accessibility checks (WCAG 2.0 Level AA)
- All images have alt attributes with descriptive text
- All form inputs have associated labels
- Data tables have caption and proper th/scope attributes
- Colour contrast meets 4.5:1 ratio for normal text
- Links have descriptive text (flag "click here" or "read more")
- Language attribute is set on the html element

### WET-BOEW component checks
- Expand/collapse uses native details/summary elements
- Tables use class="provisional gc-table table" for responsive behaviour
- Alerts use section element with the correct alert classes
- Interactive checkboxes use class="provisional gc-chckbxrdio"

### Report format
List each issue with:
- Severity (error, warning, info)
- Line number
- Description of the problem
- How to fix it
  • GC accessibility audit: WCAG 2.0 Level AA compliance check

    This skill performs a focused accessibility review of a GC web page.

    Create the file .claude/skills/a11y-audit/SKILL.md:

    ---
name: a11y-audit
description: Audit web pages for WCAG 2.0 Level AA accessibility compliance as required by the Government of Canada Standard on Web Accessibility.
argument-hint: "[filename]"
context: fork
agent: Explore
allowed-tools: Read, Grep, Glob
---

## Accessibility audit

Perform a WCAG 2.0 Level AA audit on $ARGUMENTS.

### Check each of the following

1. **Headings**: Proper hierarchy (h1 then h2 then h3, no skipped levels)
2. **Images**: All img elements have descriptive alt attributes
3. **Links**: Link text is meaningful out of context (not "click here" or "learn more")
4. **Colour**: Text meets 4.5:1 contrast ratio (check inline styles and classes)
5. **Forms**: All inputs have labels, fieldsets group related controls, error messages are clear
6. **Tables**: Data tables have captions, th elements with scope, and data-label attributes for responsive tables
7. **Language**: html element has lang attribute, bilingual content has proper lang attributes on each section
8. **Keyboard**: No elements rely solely on mouse events (onmouseover without onfocus)
9. **ARIA**: ARIA roles and attributes are used correctly (no redundant roles on semantic elements)

### Output

Provide a summary table listing:
- Check category
- Pass/Fail status
- Number of issues found
- Specific elements and line numbers for each failure
  • GC web patterns: background knowledge for WET-BOEW components

    This skill provides Claude with reference knowledge about GCWeb patterns. It loads automatically when you work on GC web content, but cannot be invoked with a slash command.

    Create the file .claude/skills/gc-web-patterns/SKILL.md:

    ---
name: gc-web-patterns
description: GCWeb design patterns and WET-BOEW component reference for Canada.ca pages. Covers expand/collapse, alerts, tables, buttons, checkboxes, and grid layout.
user-invocable: false
---

## GCWeb component reference

### Expand/collapse (details/summary)

Single:
```html
<details>
    <summary>Title</summary>
    <p>Hidden content</p>
</details>
```

Multiple in a list:
```html
<ul class="list-unstyled">
    <li>
        <details>
            <summary>Item 1</summary>
            <p>Content</p>
        </details>
    </li>
</ul>
```

### Alerts

```html
<section class="alert alert-info">
    <h3>Information</h3>
    <p>Alert content</p>
</section>
```

Types: alert-danger, alert-warning, alert-info, alert-success.

### Responsive tables

```html
<table class="provisional gc-table table" id="myTable">
    <caption>Table description</caption>
    <thead>
        <tr>
            <th>Column</th>
        </tr>
    </thead>
    <tbody>
        <tr>
            <td data-label="Column">Value</td>
        </tr>
    </tbody>
</table>
```

### Buttons

- Super-task: class="btn btn-call-to-action"
- Primary: class="btn btn-primary"
- Secondary: class="btn btn-default"
- Danger: class="btn btn-danger"

### Grid layout (Bootstrap 3)

```html
<div class="container">
    <div class="row">
        <div class="col-md-6">Column 1</div>
        <div class="col-md-6">Column 2</div>
    </div>
</div>
```

### Interactive checkboxes

```html
<fieldset class="provisional gc-chckbxrdio">
    <legend>Question:</legend>
    <ul class="list-unstyled lst-spcd-2">
        <li class="checkbox">
            <input type="checkbox" id="chk1">
            <label for="chk1">Option 1</label>
        </li>
    </ul>
</fieldset>
```

Best practices

Do

  • Write specific, keyword-rich descriptions so Claude matches the right skill automatically
  • Keep SKILL.md under 500 lines and move detailed reference material to supporting files
  • Set disable-model-invocation: true for any skill that modifies files, deploys, or commits
  • Include argument-hint for skills that accept arguments
  • Store skills in version control so the team shares the same standards
  • Include examples of correct output in your instructions so Claude understands what you expect
  • Test each skill by invoking it with /skill-name before relying on it

Avoid

  • Vague descriptions like "helpful tool" — Claude cannot match these to your prompts
  • Putting all instructions in one giant SKILL.md — split large reference material into supporting files
  • Allowing automatic invocation for skills that have side effects (use disable-model-invocation: true)
  • Creating skills for concepts Claude already understands well — focus on your team's specific standards and patterns
  • Forgetting to update the description when the skill's purpose changes

Getting started

To begin using skills in your project, create the .claude/skills/ directory and add your first SKILL.md file. Start with one of the GC examples above and customize it for your team's workflow.
